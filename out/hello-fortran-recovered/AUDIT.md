# Audit — hello_gfortran_O2

Audit of hello_gfortran_O2 reveals a significant gap between binary metadata and recovered function set. The binary declares 7 functions and 14 imports, but only 2 functions were recovered (main and hello_program_main). The 5 missing functions are likely Fortran runtime stubs or CRT scaffolding (frame_dummy, register_tm_clones, __do_global_dtors_aux, _start, deregister_tm_clones) silently dropped. The binary reports 0 strings, which is suspicious for a 'hello' program — the literal greeting should be present in .rodata. The hello_program_main function carries 9 rewriter assumptions on a tiny Fortran-emitted routine, indicating low-confidence reconstruction of the libgfortran I/O sequence. Module decomposition splits a trivial program across two C files, which does not reflect the original single-file Fortran source.

- **[high]** confidence_gap @ src/core.c::hello_program_main: Confidence 0.72 with 9 rewriter assumptions on what should be a ~10-instruction routine (Fortran MAIN__ calling libgfortran to print one string). 9 assumptions on so little code suggests the rewriter guessed at the dtparm struct layout, character-length argument, unit number, and filename/line metadata passed to _gfortran_st_write. At -O2, gfortran also inlines/reorders dtparm initialization, which the rewriter likely could not faithfully reproduce.  → _Pin libgfortran ABI signatures from a known gfortran version, regenerate hello_program_main against those fixed prototypes, and drive the assumption count toward zero._
- **[medium]** assumption_risk @ project (strings_count=0): Binary reports strings_count=0, yet a Fortran 'hello' program must contain the greeting literal plus typically a source-filename string passed to _gfortran_st_write for runtime error reporting. A zero string count indicates the string-extraction stage failed or skipped .rodata.str* sections. Consequently the recovered hello_program_main cannot have the actual printed text wired up correctly.  → _Re-scan .rodata and .rodata.str* sections; ensure string literals are surfaced and threaded into hello_program_main's transfer_character_write call._
- **[medium]** module_coherence @ project (main.c + src/core.c split): Original source is a single Fortran program unit. Splitting it into main.c (C runtime entry) and src/core.c (renamed MAIN__) is partly defensible because gfortran does emit a separate main that calls MAIN__, but renaming MAIN__ to 'hello_program_main' loses the gfortran name-mangling contract (MAIN__). Anything linking against the original mangled symbol will fail.  → _Keep the original symbol name MAIN__ (with an alias if a friendlier C name is desired) and document the rename so the call site in main.c matches the definition._
- **[medium]** signature_mismatch @ main.c -> src/core.c::hello_program_main: The C runtime stub emitted by gfortran calls MAIN__ with no arguments and void return, after first invoking _gfortran_set_args(argc, argv) and _gfortran_set_options(...). Given main.c carries 4 assumptions and the runtime helpers are missing entirely, there is meaningful risk that main forwards argc/argv directly to hello_program_main rather than to _gfortran_set_args, producing a signature mismatch at the call boundary.  → _Verify main's prototype is `int main(int argc, char **argv)` and that it calls _gfortran_set_args/_set_options before invoking a zero-arg hello_program_main(); flag any direct argc/argv pass-through to MAIN__ as a rewrite bug._
- **[low]** missing_error_path @ src/core.c::hello_program_main: libgfortran I/O sequences wrap st_write/transfer/st_write_done; if any sets the dtparm common error flags, gfortran-emitted code branches to error reporting. With 5 functions missing and no error handling visible in the 2-function recovery, the rewrite has likely silenced these post-call status checks. Benign at runtime for a hello-world but a faithfulness gap.  → _Restore the post-call status checks around _gfortran_st_write_done, even if they appear dead at -O2._
- **[low]** dead_code @ project (imports vs callers): 14 imports against only 2 recovered functions implies most imports have no visible caller in the rewritten tree. Either the recovered functions silently omit calls to those imports, or the imports are CRT scaffolding (__libc_start_main, _init, __cxa_finalize, __gmon_start__) that legitimately is not called from user code. Cannot disambiguate without the import list.  → _Cross-reference each of the 14 imports to a call site in main or hello_program_main; flag any non-CRT import with zero references for review._
